# **TRAVEL PLANNER**
## High-Level Design Document

Version 5.0  —  May 2026

---

# **1\. Overview**

This document describes the high-level architecture of a collaborative AI-powered travel planner. The system allows users to input a destination, travel dates, and budget, and receive AI-generated suggestions for places to visit, estimated flight costs, and hotel/hostel options.

Key design principles guiding this architecture:

* Client-side AI inference — users supply their own AI provider API keys, which are stored locally in the browser and never transmitted to the server.

* Serverless backend — Firebase replaces a custom backend, providing authentication, database (Firestore), and real-time capabilities out of the box.

* Collaborative by design — trips can be shared with groups via invite links, with real-time synchronisation of suggestions across members.

* Zero external data APIs — flight and hotel suggestions are AI-generated estimates, not live data, keeping the system dependency-free.

# **2\. Architecture Overview**

The system is divided into two primary zones: the client (browser-based React application) and the Firebase backend. There is no custom application server.

| Client Layer — Browser / React App |  |  |
| :---- | :---- | :---- |
| **Input Form** | Collects destination, dates, budget, and travel interests from the user. | React \+ Tailwind |
| **API Key Store** | Persists the user's AI provider key in localStorage. Never sent to any server. | localStorage |
| **AI Orchestrator** | Builds structured prompts and fans out parallel calls to the AI provider for places, flights, and hotels. | JS fetch / SDK |
| **AI Provider** | The user's chosen LLM (OpenAI, Anthropic, or Gemini) processes prompts and returns structured suggestions. | User's API key |
| **Result Panels** | Three panels display AI-generated suggestions: Places, Flights, and Hotels. | React components |
| **Follow-up Chat** | A persistent chat interface for refining suggestions, asking follow-up questions, or replanning. | React \+ AI SDK |

| Firebase Layer — Auth, DB, Real-time |                                                                                                                        |  |
|:-------------------------------------|:-----------------------------------------------------------------------------------------------------------------------| :---- |
| **Auth**                             | Handles user identity via magic link (passwordless email) and Google OAuth. Issues JWTs used by all Firebase services. | Firebase Auth |
| **Firestore DB**                     | NoSQL document store for trips, AI-generated suggestions, and group membership.                                        | Firebase Firestore |
| **Realtime**                         | Firestore's native `onSnapshot` listeners broadcast changes instantly to all connected clients.                        | Firestore SDK |
| **Security Rules**                   | Firestore Security Rules enforce that only authenticated trip members can read or write trip data — enforced at the DB layer. | Firebase Rules |
| **JS Client SDK**                    | Single npm package providing a unified interface to all Firebase services from the browser.                            | Firebase-js |

# **3\. Data Flow**

## **3.1 Generating suggestions**

* User fills in the input form (destination, dates, budget, interests) and clicks Generate.

* The AI orchestrator retrieves the user's API key from localStorage.

* Three parallel requests are fired to the chosen AI provider: one each for Places, Flights, and Hotels.

* Responses are parsed and displayed in the three result panels.

* User clicks Save Trip — the trip metadata and all three suggestion sets are written to Firestore via the JS SDK.

## **3.2 Sharing a trip**

* The trip owner clicks Share — the Share Service generates a unique invite link containing the trip ID.

* The recipient opens the link. If unauthenticated, Firebase Auth prompts login via magic link or Google OAuth.

* On login, the recipient is added to the trip_members collection with a viewer or collaborator role.

* Firebase security rules grant read (and optionally write) access based on membership.

## **3.3 Real-time collaboration**

* Any authenticated trip member's client subscribes to the Firestore document or collection for that trip's data.

* When a collaborator adds or edits a suggestion, the Firestore change event is broadcast instantly to all subscribers.

* No custom WebSocket server is needed — Firebase manages the connection.

# **5\. Security Rules**

Firestore Security Rules are defined in the Firebase Console and enforced at the database layer. This ensures data isolation even if client-side logic is bypassed.

**trips collection**

* read — allowed if `request.auth.uid == resource.data.owner_id` OR user is in `trip_members` for this `trip_id`.

* create — allowed if `request.auth.uid != null`.

* update / delete — allowed if `request.auth.uid == resource.data.owner_id`.

**suggestions collection**

* read — allowed if user is a member of the parent trip.

* create / update — allowed if user is a collaborator or owner of the parent trip.

* delete — allowed if user is the owner of the parent trip.

**trip_members collection**

* read — allowed if user is a member of the trip.

* create — only the trip owner can add members.

# **7\. Key Design Decisions**

**Why Cloudflare Pages for deployment?**

Cloudflare Pages serves static assets from a global edge network (300+ locations) with zero cold starts and no server to maintain. Critically, because all AI keys live in the user's localStorage and Firebase handles all backend concerns, there are no application secrets stored in the deployment platform at all — eliminating the primary risk vector seen in recent platform breaches. Cloudflare's security posture and infrastructure maturity make it the strongest choice for a zero-backend-secret frontend.

**Why client-side AI?**

Keeping AI calls in the browser means the server never sees user API keys or prompt data. It also means infrastructure costs scale with the user's own quota, not yours. The tradeoff is that AI keys are visible in browser memory — mitigated by storing only in localStorage and never logging or transmitting them.

**Why Firebase (Firestore) instead of a custom backend?**

A custom Express or FastAPI server would require deploying, maintaining, and securing separate auth, database, and WebSocket services. Firebase (specifically Firestore) collapses all three into a single managed platform. The JS client SDK provides a single import for the frontend. For a project at this scale, the developer velocity provided by Firestore strongly favours this choice over a relational database.

**Why AI-generated estimates instead of live flight/hotel APIs?**

Live APIs (Amadeus, Booking.com) require API keys, introduce external dependencies, and have rate limits and costs. LLMs have broad knowledge of typical travel costs and can provide useful ballpark figures — clearly labelled as estimates. This keeps the system fully self-contained and avoids data freshness concerns.

**Why Map/Documents for suggestions?**

The structure of a 'place' suggestion differs from a 'flight' or 'hotel' suggestion. Using flexible document schemas in Firestore allows each type to carry its own data without requiring multiple collections or complex migrations.

# **8\. Future Considerations**

* **Allow users to drag suggestions into a day-by-day itinerary view, stored as an itinerary sub-collection in Firestore.:** Itinerary builder

* **Notify group members when new suggestions are added, using Firebase Cloud Functions + web push.:** Push notifications

* **Generate a printable trip summary from the saved suggestions.:** Export to PDF

* **Store the prompts used to generate each suggestion alongside the output, so users can regenerate with a different AI provider.:** Prompt versioning

* **Add Firebase Cloud Functions as a thin middleware layer if abuse prevention becomes necessary.:** Rate limiting
